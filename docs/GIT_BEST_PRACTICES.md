# Лучшие практики Git для проекта

Документ описывает структуру проекта и настройки Git, соответствующие общепринятым практикам.

## Структура проекта

```
graduate-work/
├── CMakeLists.txt          # Корневая конфигурация CMake
├── .gitignore              # Игнорируемые файлы
├── .clang-format           # Форматирование кода
├── .clang-tidy             # Статический анализ
├── README.md               # Главная документация
├── SETUP.md                # Инструкции по установке
├── include/                # Публичные заголовки
│   └── graph_reliability/
├── src/                    # Исходный код
├── tests/                  # Unit-тесты
├── graphs_data/            # Тестовые графы (версионируются)
├── docs/                   # Документация
│   ├── references/        # Научные источники
│   │   └── papers/         # PDF (опционально в репо)
│   └── ...
└── build/                  # Сборка (НЕ в Git)
```

### Соответствие лучшим практикам

| Практика | Реализация |
|----------|------------|
| **Разделение исходников и заголовков** | `src/` и `include/` — стандартная структура C++ |
| **Out-of-source build** | Сборка в `build/`, не в корне проекта |
| **Тесты рядом с кодом** | `tests/` с CMakeLists.txt, Google Test |
| **Данные отдельно** | `graphs_data/` для входных графов |
| **Документация в репо** | `docs/` — архитектура, алгоритмы, руководства |
| **Конфигурация форматирования** | `.clang-format`, `.clang-tidy` версионируются |

## .gitignore: что игнорировать

### 1. Артефакты сборки

- `build/`, `bin/`, `lib/`, `out/` — каталоги сборки
- `CMakeCache.txt`, `CMakeFiles/`, `cmake_install.cmake`, `Makefile`
- `*.o`, `*.obj`, `*.a`, `*.lib`, `*.dll`, `*.so`, `*.exe`
- `_deps/` — FetchContent (googletest, spdlog) — внутри `build/`, уже игнорируется

**Почему:** Артефакты можно пересобрать из исходников. Их версионирование засоряет историю и создаёт конфликты.

### 2. IDE и редакторы

- `.vscode/`, `.idea/`, `.cursor/`, `.vs/`
- `*.user`, `*.vcxproj.user`
- `*.swp`, `*.swo`, `*~` — временные файлы vim/emacs

**Почему:** Настройки IDE часто персональные и платформо-зависимые. Исключение: общие настройки (например, рекомендуемые расширения) можно хранить в `.vscode/extensions.json` и не игнорировать.

### 3. Системные файлы

- `.DS_Store`, `Thumbs.db` — метаданные ОС
- `*.tmp`, `*.temp`, `*.bak`, `*.backup`

**Почему:** Не относятся к коду и создают шум в репозитории.

### 4. Результаты тестов и логи

- `test_results.csv`, `test_output.csv`, `results*.csv`
- `*.log`, `coverage/`, `*.gcov`, `*.gcno`, `*.gcda`

**Почему:** Генерируются при запуске. Для воспроизводимости лучше описать команды в README/SETUP, чем хранить результаты.

### 5. Документация (сгенерированная)

- `docs/html/`, `docs/latex/`, `docs/xml/`, `docs/man/` — вывод Doxygen

**Почему:** Создаётся из исходников. При необходимости можно генерировать при CI/CD.

### 6. Секреты и окружение

- `.env`, `.env.local`, `*.pem`, `*.key` (если появятся)

**Почему:** Не должны попадать в репозиторий. Использовать переменные окружения или секреты CI.

## Что версионировать

| Категория | Примеры |
|-----------|---------|
| **Исходный код** | `src/*.cpp`, `include/**/*.h` |
| **Конфигурация сборки** | `CMakeLists.txt`, `tests/CMakeLists.txt` |
| **Тестовые данные** | `graphs_data/*.txt`, `tests/test_config.json` |
| **Документация** | `docs/*.md`, `README.md`, `SETUP.md` |
| **Стиль и анализ** | `.clang-format`, `.clang-tidy` |
| **Скрипты** | `check_environment.ps1` (если есть) |

## PDF в docs/references/papers/

**Текущее состояние:** PDF-файлы научных работ находятся в репозитории.

**Варианты:**

1. **Оставить в репо** — удобно для воспроизводимости и офлайн-работы. Минус: увеличение размера репо.
2. **Игнорировать** — добавить `docs/references/papers/*.pdf` в `.gitignore`, хранить PDF отдельно (облако, другой репо). В `docs/references/README.md` указать, где их взять.
3. **Git LFS** — для больших бинарных файлов: `git lfs track "*.pdf"`. Требует настройки LFS на сервере.

Для выпускной работы часто выбирают вариант 1, если размер репо приемлем.

## Рекомендации перед push

1. **Проверить статус:** `git status` — нет ли лишних файлов.
2. **Собрать проект:** `cmake --build build` — сборка без ошибок.
3. **Запустить тесты:** `./build/graph_reliability_tests` или `ctest`.
4. **Проверить .gitignore:** `git check-ignore -v <файл>` — убедиться, что нужные файлы игнорируются.
5. **Не коммитить:** `build/`, `*.log`, `results*.csv`, личные настройки IDE.

## Полезные команды

```bash
# Что будет закоммичено
git status

# Проверить, игнорируется ли файл
git check-ignore -v results.csv

# Удалить из индекса уже добавленный файл (оставить на диске)
git rm --cached results.csv
```
